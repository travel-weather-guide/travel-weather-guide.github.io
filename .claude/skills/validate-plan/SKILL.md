# /validate-plan

PLAN-CURRENT.md에 작성된 현재 플랜을 다각도로 검증한다.

## 사용법
```
/validate-plan
```

## 워크플로우

### 1단계: 플랜 로드
1. `PLAN-CURRENT.md`를 읽는다
2. 진행 중인 항목이 없으면 안내하고 종료
3. `PLAN.md`, `CLAUDE.md`, `index.html`도 읽어 프로젝트 맥락을 파악한다

### 2단계: 검증 수행

아래 5가지 관점에서 플랜을 검증한다. 각 항목마다 **통과/주의/문제** 중 하나로 판정.

#### A. 기술적 타당성
- 플랜에 기술적으로 무리한 부분이 없는가?
- 의존하는 외부 API나 서비스가 안정적인가?
- 알려진 기술적 제약이나 함정이 있는가?

#### B. 구현 최적성
- 이 접근법이 현재 알려진 best practice인가?
- 더 나은 방법이나 패턴이 존재하는가? (웹 검색으로 최신 사례 확인)
- 기존 코드와 자연스럽게 합쳐지는가, 아니면 구조를 크게 바꿔야 하는가?

#### C. 오버엔지니어링 체크
- 당장 필요하지 않은 기능을 미리 설계하고 있지 않은가?
- 추상화 레이어나 설정 옵션을 과도하게 만들고 있지 않은가?
- 한 번에 너무 많은 것을 바꾸려 하지 않는가?

#### D. 추가 비용
- 유료 외부 API나 서비스를 새로 결제해야 하는가?
- 무료 티어로 커버되는가, 유료 플랜이 필요한가?

#### E. CLAUDE.md 정합성
- CLAUDE.md에 명시된 프로젝트 원칙이나 작업 규칙을 위반하지 않는가?
- 기존 코드 패턴/구조와 일관되는가?

### 3단계: 결과 보고

아래 형식으로 사용자에게 보고한다:

```
## 플랜 검증 결과: {항목명}

| 검증 항목 | 판정 | 요약 |
|-----------|------|------|
| 기술적 타당성 | 통과/주의/문제 | 한 줄 요약 |
| 구현 최적성 | 통과/주의/문제 | 한 줄 요약 |
| 오버엔지니어링 | 통과/주의/문제 | 한 줄 요약 |
| 비용/리소스 | 통과/주의/문제 | 한 줄 요약 |
| 프로젝트 정합성 | 통과/주의/문제 | 한 줄 요약 |

### 상세

(주의/문제 항목에 대해서만 구체적 설명과 개선 제안)

### 결론

(전체 판정: "진행 OK" / "일부 수정 후 진행" / "재검토 필요")
(수정이 필요하면 구체적으로 무엇을 어떻게 바꿀지 제안)
```

### 4단계: 수정 반영 (선택)
- 사용자가 수정을 원하면 PLAN-CURRENT.md를 직접 업데이트한다
- 체크리스트 축소, 항목 제거, 접근법 변경 등을 반영
